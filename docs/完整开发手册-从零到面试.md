## 1. 面试怎么讲这个项目

### 1.1 一句话介绍

**基于多智能体协作的 AI 图文创作平台**：用户输入选题 → 5+1 个 Agent 生成标题、大纲、正文、配图 → 人机协作三阶段确认 → 导出 Markdown；含 VIP 会员与 Stripe 支付。

### 1.2 30 秒版（电梯演讲）

> 我做了一个 AI 文章创作器，后端 FastAPI + MySQL + Redis，前端 Vue 3。核心是**多阶段状态机 + 多 Agent 编排**：阶段 1 生成标题让用户选，阶段 2 流式生成大纲可编辑，阶段 3 生成正文、审核、配图、合并。进度通过 **SSE** 实时推送到前端。配图用**策略模式**支持 Pexels、Mermaid、AI 生图等 6 种方式。会员体系对接 **Stripe Checkout + Webhook**，Session 用 Redis 存，Cookie 只带 uuid。

### 1.3 2 分钟版（STAR 结构）

| 部分 | 内容 |
|------|------|
| **背景 S** | 传统写长文耗时长；课程要求对标 Spring AI 多 Agent 编排 |
| **任务 T** | 实现 Python 版全栈：登录、配额、三阶段创作、配图、VIP 支付、管理统计 |
| **行动 A** | 分层架构 Router→Service→Agent；asyncio 长任务；SSE 缓冲与断线回补；Stripe metadata 关联用户；Session 与 DB 双源刷新 |
| **结果 R** | 完整跑通创作流水线；本地 Stripe 测试支付升 VIP；agent_log 可追踪每个智能体 |

### 1.4 可强调的技术亮点（面试加分）

1. **多智能体编排**：`orchestrator.py` 按 phase 调度 Title→Outline→Content→Reviewer→Image→Merge  
2. **SSE 实时通信**：大纲/正文流式输出 + 配图进度；`sse_manager` 消息缓冲防丢  
3. **策略模式配图**：`image_service_strategy.py` 注册 6 种 `ImageSearchService`  
4. **人机协作状态机**：`article.phase`：TITLE_SELECTION → OUTLINE_EDITING → CONTENT_GENERATION → COMPLETED  
5. **Session 鉴权**：Cookie 只存 uuid，Redis 存用户快照，支付后从 MySQL 刷新 role  
6. **异步长任务**：`article_async_service` 后台跑 phase，**禁止 uvicorn --reload** 否则任务被杀

### 5.3 登录流程（数据示例）

```

2. user_service 查 MySQL，校验密码

6. 返回 LoginUserVO 给前端
```

## 8. 第 5 阶段：VIP 支付（Stripe）

> 详细实战见 [2026-06-01-Git与Stripe实战总结.md](./2026-06-01-Git与Stripe实战总结.md)

### 8.1 流程摘要

```text
前端 POST /payment/create-vip-session
  → Cookie SESSION → Redis → user.id
  → stripe.checkout.Session.create(metadata.userId)
  → INSERT payment_record PENDING
  → 返回 checkout URL → 前端跳转 Stripe

用户付款 → stripe listen → POST /webhook/stripe
  → handle_payment_success
  → payment_record SUCCEEDED + user.userRole=vip

跳回 /vip?success=true → fetchLoginUser → deps 从 MySQL 刷新 Session
```

### 8.2 本地要不要启动 Stripe？——要，才能**自动**升 VIP

| 进程 | 不启动会怎样 |
|------|----------------|
| 后端 uvicorn | 无法创建支付链接 |
| 前端 `npm run dev` | 无法打开 VIP 页 |
| **`stripe listen`（Stripe CLI）** | 能付款，但 **Webhook 收不到** → 数据库仍是 `user`，页面仍显示「升级 VIP」 |

**结论：** 本地开发要 **同时开三个终端**；测 VIP 支付时 **`stripe listen` 必须一直运行**（不要关）。

#### 第一次使用：登录 Stripe CLI

```powershell
# 若 PATH 已配置 stripe 命令
stripe login

# Windows 若提示找不到命令，用完整路径（按你解压目录修改）
D:\stripe_1.42.1_windows_x86_64\stripe.exe login
```

浏览器完成授权即可。

#### 日常测试 VIP：启动 Webhook 转发（保持运行）

```powershell
# 后端端口须与 uvicorn 一致（本仓库常用 8569）
stripe listen --forward-to localhost:8569/api/webhook/stripe

# Windows 完整路径示例
D:\stripe_1.42.1_windows_x86_64\stripe.exe listen --forward-to localhost:8569/api/webhook/stripe
```

终端会输出类似：

```text
Your webhook signing secret is whsec_xxxxxxxx...
```

**把 `whsec_...` 写入** `python-backend/.env`：

```env
STRIPE_WEBHOOK_SECRET=whsec_xxxxxxxx
```

然后 **重启后端**。注意：`stripe listen` **每次重启会生成新的 whsec**，变了就要更新 `.env` 并重启后端。

## 14. 面试高频问题与参考答案

### Q1：为什么用 SSE 不用 WebSocket？

> 创作进度是**服务端单向推送**，SSE 基于 HTTP，与现有 Cookie 鉴权、Vite 代理兼容好；WebSocket 双向能力用不上，部署也更复杂。

### Q2：Session 怎么实现的？

> 登录后生成 uuid 写 Cookie；Redis 存 `session:{uuid}` → JSON 用户信息。Cookie 不含敏感字段。支付升 VIP 后从 MySQL 刷新 role 写回 Redis。

### Q3：多 Agent 怎么编排的？

> `orchestrator` 按 article.phase 调用不同 agent 模块；phase3 内顺序：正文→Reviewer→配图分析→并行配图→合并；异步由 `article_async_service` 的 asyncio 任务执行。

### Q4：配额和 VIP 怎么控制？

> 普通用户 `quota` 字段，创作时扣减；VIP/admin 在 `hasQuota` 和 `_is_vip_or_admin` 中无限；高级配图前后端双重校验。

### Q5：Stripe 支付如何保证用户和订单对应？

> Checkout Session 的 `metadata.userId`；Webhook 里用 `stripeSessionId` 查 `payment_record` 再更新对应 user。

### Q6：如果 LLM 返回非法 JSON 怎么办？

> `llm_retry.py`：重试 + 从 markdown 代码块提取 JSON；失败写 agent_log 并 SSE ERROR。

### Q7：配图失败会怎样？

> 策略内 catch；Mermaid 可 Kroki 降级；最终 `PICSUM` 随机图保底，不中断全文生成。

### Q8：你负责哪部分 / 难点是什么？

> （按你真实经历答）例如：SSE 断线回补、Stripe Webhook metadata 解析 bug、Session 与 VIP 状态不一致修复。
